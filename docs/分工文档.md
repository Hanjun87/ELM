# ELM 外卖平台 — 小组分工文档

**项目名称**: ELM 外卖平台（多角色外卖配送系统）
**小组成员**: 吴含俊、尹晓涵、张嘉溟、李嘉豪（共 4 人）
**分工原则**: 吴含俊独立负责全部前端（4 个角色端），后端开发、数据库设计、测试、文档、部署等其余工作由尹晓涵、张嘉溟、李嘉豪三人平均分配

## 二、吴含俊 — 前端全栈开发

**负责范围**：Customer、Merchant、Rider、Manager 四个角色端的全部前端界面与逻辑，共 4 个独立 React 应用。

### 2.1 技术选型与工程搭建
- 4 个应用均采用 React 19 + TypeScript + Vite + Tailwind CSS v4
- 搭建 axios 请求层（`api/config.ts`：统一请求拦截器自动附加 JWT、响应拦截器处理 401 失效登出；`api/index.ts`：按业务域拆分的接口调用模块）
- 搭建 `AuthContext`（JWT 登录状态管理、角色白名单校验、本地存储持久化登录态）
- 4 个应用的 `Dockerfile`（Node 多阶段构建 + Nginx 静态资源服务 + SPA 路由回退）

### 2.2 Customer 端（客户端）
- 登录/注册页面
- 商家列表、商家详情、商品浏览（按分类筛选）
- 购物车、下单结算、支付确认
- 订单列表、订单详情、取消订单
- 收货地址管理（新增/编辑/删除/设默认）
- 商品评价提交
- 优惠券领取与查看
- 修复 `BottomNav` 组件遗留的死代码引用与底部导航点击失效问题

### 2.3 Merchant 端（商家端）
- 登录页面（角色校验：仅允许商家账号登录）
- 工作台看板（实时订单概览、快捷操作入口）
- 订单管理（接单/拒单/确认出餐/出餐完成，状态流转可视化）
- 商品管理（新增/编辑/上下架/删除，分类选择）
- 店铺设置（店铺信息编辑、一键开关店）
- 评价管理（查看评价、商家回复）
- 将原有 Mock 数据（`store.ts`）全部改造为真实接口对接

### 2.4 Rider 端（骑手端）
- 登录页面（角色校验：仅允许骑手账号登录）
- 工作台（可接单大厅、工作状态切换：空闲/忙碌/离线）
- 任务面板（进行中/待接单/已完成三个分栏，抢单/确认取餐/确认送达操作）
- 个人中心（骑手档案展示：姓名、评分、余额、累计单量、退出登录）
- 异常上报交互（UI 层，含图片说明占位）

### 2.5 Manager 端（管理端）
- 登录页面（角色校验：仅允许管理员账号登录）
- 数据看板（GMV、订单数、商家数、用户数、订单状态分布，对接真实管理后台接口）
- 用户管理（列表筛选、封禁/解封操作）
- 商家审核、财务结算、内容运营（轮播图/公告）三个模块的界面搭建

## 三、尹晓涵 — 账号认证 + 地址/骑手模块

**负责范围**：用户身份体系与两个外围业务模块，是整个系统权限校验的基础。

### 3.1 accounts（账号认证模块）
- `User` / `Role` / `UserRole` 数据模型设计（自定义用户模型，手机号作为登录凭证）
- 注册接口（角色白名单校验：仅允许 customer/merchant/rider 自助注册，管理员账号不可通过接口自助创建）
- 登录接口（JWT access/refresh token 签发，账号封禁状态校验）
- 当前用户信息接口
- RBAC 权限体系落地（`has_role()` 权限判断方法）
- 编写 accounts 模块单元测试（注册成功/重复注册/非法角色/登录成功/密码错误/用户不存在等场景）

### 3.2 addresses（收货地址模块）
- `Address` 数据模型设计
- 地址增删改查接口
- 默认地址设置接口（原子事务保证同一用户仅有一个默认地址）
- 编写 addresses 模块单元测试

### 3.3 riders（骑手档案模块）
- `Rider` 数据模型设计（工作状态枚举：离线/空闲/忙碌/配送中）
- 骑手个人信息查看/更新接口
- 骑手工作状态切换接口
- 编写 riders 模块单元测试

### 3.4 文档
- 《需求说明书》：系统角色定义、按角色划分的功能需求表、非功能需求（安全/数据完整性）、已知功能缺口梳理
- 《03-RBAC权限设计》文档：角色定义、权限矩阵、数据隔离策略

## 四、张嘉溟 — 核心交易模块（商家/商品/订单）+ 数据库设计

**负责范围**：系统最核心、最复杂的业务链路——从商家开店、商品上架到订单完整生命周期，是三人中工作量最集中的部分。

### 4.1 merchants（商家模块）
- `Merchant` 数据模型设计
- 商家列表/详情公开接口
- 商家端「我的店铺」查看与编辑接口
- 开关店接口
- 修复商家可自行篡改评分/月销量字段的数据完整性问题（设置只读字段）
- 编写 merchants 模块单元测试

### 4.2 products（商品模块）
- `Product` / `Category` 数据模型设计
- 商品公开浏览接口（按分类筛选、库存/上架状态过滤）
- 商家端商品管理接口（创建/更新/上下架/删除）
- 价格与库存合法性校验（价格 ≥ 0.01，库存 ≥ 0，防止负数攻击）
- 编写 products 模块单元测试（含商家数据隔离性测试：验证商家 A 无法操作商家 B 的商品）

### 4.3 orders（订单核心模块，本项目工作量最大的模块）
- `Order` 数据模型设计（含地址快照、商品明细快照、9 种状态枚举、多个时间戳字段）
- 客户端接口：下单（服务端价格计算、库存原子扣减、起送金额校验）、支付、取消订单
- 商家端接口：订单列表、接单、拒单、确认出餐、出餐完成（四段状态流转）
- 骑手端接口：可接单列表、我的配送单、抢单（并发锁防超卖）、确认取餐、确认送达
- 取消/拒单后的库库存与销量自动回滚逻辑
- 编写 orders 模块单元测试（本项目测试用例最多的模块，覆盖下单校验、库存回滚、状态流转全链路）

### 4.4 数据库设计
- 全系统数据库 E-R 图设计（13 张核心表的实体关系，使用 Mermaid 绘制）
- 各表字段详细设计文档（字段类型、约束、默认值、业务含义说明）
- 《04-订单生命周期设计》文档：状态机图、状态流转触发条件与后置动作说明

## 六、协作说明

1. **接口约定先行**：后端三人（尹晓涵、张嘉溟、李嘉豪）在各自模块开发前，先与吴含俊确认接口的请求/响应字段格式，避免前后端返工。
2. **统一响应规范**：全部接口遵循 `{code, message, data}` 统一响应格式，错误码按模块分段（1xxx 账号、3xxx 商家商品、4xxx 订单、5xxx 骑手/优惠券、6xxx 评价、9xxx 通用参数错误）。
3. **测试驱动**：每个后端模块开发者负责编写并维护自己模块的单元测试，合并前保证 `python manage.py test` 全部通过。
4. **文档同步更新**：功能变更后，对应负责人同步更新《API接口文档》与《需求/设计说明书》，避免文档与实现脱节。
