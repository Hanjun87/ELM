# ELM 外卖平台 — 需求说明书

---

## 一、项目背景

ELM 是一个面向本地生活服务的外卖配送平台，支持餐厅在线接单、骑手即时配送、客户手机下单，以及平台方对商家、用户和订单进行集中管理。系统采用前后端分离架构，提供四个独立的角色端界面。

---

## 二、系统角色

| 角色   | 英文标识 | 说明                                           |
| ------ | -------- | ---------------------------------------------- |
| 客户   | customer | 通过 Customer 端进行商品浏览、下单、支付、评价 |
| 商家   | merchant | 通过 Merchant 端管理店铺、商品、订单、查看评价 |
| 骑手   | rider    | 通过 Rider 端接单、配送、更新状态              |
| 管理员 | admin    | 通过 Manager 端管理用户、商家、查看平台数据    |

### 2.1 客户（C 端用户）

客户是系统的流量入口与订单产生端，核心诉求是流畅的浏览体验与便捷的下单流程。主要能力包括：

- 账户与个人中心：手机号注册 / 登录、密码找回；管理个人收货地址、查看历史订单、管理收藏与优惠券。
- 商品浏览与搜索：按分类、距离、评分等维度筛选商家与商品；关键词模糊搜索；查看商品详情（图文、规格、评价）。
- 购物车与下单结算：商品加入购物车、修改数量、选择规格；结算时选择地址、使用优惠券、在线支付。
- 订单履约与售后：实时追踪订单状态；申请退款 / 退货、联系客服、对已完成订单进行图文评价。

### 2.2 商家（B 端商户）

商家端主要用于店铺的日常运营、商品管理与订单履约，核心诉求是高效接单与经营数据分析。主要能力包括：

- 店铺与营业管理：设置店铺基本信息（Logo、简介、营业时间、起送价、配送范围）；一键开关店、处理店铺公告。
- 商品管理：新增 / 编辑 / 删除商品，设置商品分类、价格、库存、多规格（如大小份、辣度）；批量上下架、库存预警。
- 订单处理：实时接收新订单提醒；手动 / 自动接单、打印小票、上报出餐状态、处理催单与异常订单。
- 营销与数据看板：自主配置店铺营销活动（满减、折扣、新客立减）；查看营业额、订单量、客单价、转化率等多维度经营报表。

### 2.3 骑手（履约端）

骑手是连接商家与客户的纽带，核心诉求是清晰的派单指引与便捷的订单状态流转。主要能力包括：

- 状态与接单管理：设置工作状态（开工 / 休息 / 忙碌）；接收系统派单或大厅抢单；查看待配送订单列表。
- 配送履约流转：严格按照节点更新订单状态：到店、取餐、送达；查看导航路线与预估送达时间。
- 异常与沟通处理：遇到缺货、地址错误、联系不上客户等异常情况时，一键上报异常或转单；通过系统虚拟号码或消息与客户、商家沟通。
- 收入与绩效统计：查看每日 / 每月配送单量、总收入、好评率；查看历史配送记录与提现申请。

### 2.4 管理员（平台运营端）

管理员负责平台整体运营与治理，保障交易秩序与用户体验。主要能力包括：

- 用户治理：查看平台用户列表，按角色 / 状态筛选；对违规用户进行封禁 / 解封。
- 商家管理：查看入驻商家列表，按状态筛选；审核新入驻商家、强制下架违规商家。
- 订单监控：查看平台订单流水，处理用户申诉与异常订单。
- 平台数据：查看 GMV、订单总数、商家数、用户数、取消单数等核心运营指标。
- 内容管理：管理平台公告、帮助中心、优惠券模板等公共配置。

## 四、数据库模型设计

### 4.1 用户与角色基础表

#### 1. 用户表（User）

- 用途：作为系统的基础认证表，记录所有登录系统的账号信息。
- 核心字段：用户 ID（主键）、用户名、密码哈希值、手机号、邮箱、注册时间、账号状态（正常 / 封禁）。

#### 2. 角色信息表（Role）

- 用途：定义系统中的不同身份类别，便于权限隔离。
- 核心字段：角色 ID（主键）、角色名称（如：客户、商家、配送员、管理员）、角色描述。

#### 3. 用户角色关联表（User_Role）

- 用途：实现用户与角色的多对多关系，一个用户可以拥有多个角色。
- 核心字段：关联 ID（主键）、用户 ID（外键）、角色 ID（外键）。

### 4.2 商家与商品管理表

#### 4. 商家信息表（Merchant）

- 用途：记录入驻商家的店铺详情与营业资质。
- 核心字段：商家 ID（主键）、所属用户 ID（外键，关联 User 表）、店铺名称、店铺 Logo、联系电话、详细地址、经纬度坐标、营业时间、起送金额、配送费、店铺状态（营业中 / 休息中 / 审核中）。

#### 5. 商品分类表（Category）

- 用途：对商品进行归类，便于前台检索。
- 核心字段：分类 ID（主键）、分类名称、排序权重。

#### 6. 商品表（Product）

- 用途：记录商家售卖的具体商品详情。
- 核心字段：商品 ID（主键）、所属商家 ID（外键）、所属分类 ID（外键）、商品名称、商品描述、商品主图、销售价格、当前库存、商品状态（上架 / 下架）。

### 4.3 配送员专属表

#### 7. 配送员信息表（Rider）

- 用途：记录配送员的专属业务信息与资质。
- 核心字段：配送员 ID（主键）、所属用户 ID（外键）、真实姓名、身份证号、联系电话、所属站点 / 区域、当前工作状态（空闲 / 配送中 / 休息）、账户余额。

### 4.4 订单交易核心表

#### 8. 订单主表（Order）

- 用途：记录每一笔交易的全局状态与结算信息。
- 核心字段：订单 ID（主键）、订单编号（唯一字符串）、下单客户 ID（外键）、所属商家 ID（外键）、配送员 ID（外键，可空）、订单总金额、实付金额、订单状态（待支付 / 待接单 / 待取餐 / 配送中 / 已完成 / 已取消）、下单时间、支付时间、预计送达时间。

#### 9. 订单明细表（Order_Item）

- 用途：记录订单中包含的具体商品快照，防止商品后续修改价格影响历史订单。
- 核心字段：明细 ID（主键）、所属订单 ID（外键）、商品 ID（外键）、商品名称快照、购买单价、购买数量、小计金额。

#### 10. 收货地址表（Address）

- 用途：管理客户的收货地址信息。
- 核心字段：地址 ID（主键）、所属用户 ID（外键）、收件人姓名、联系电话、省市区、详细地址、经纬度坐标、是否默认地址。

### 4.5 表结构关联说明

- 用户与商家 / 配送员：采用“基础账号 + 扩展信息”的设计。User 表负责登录认证，Merchant 和 Rider 表通过外键与 User 关联，存储各自的业务属性。
- 商家与商品：一对多关系。一个商家可以发布多个商品，Product 表通过 merchant_id 关联 Merchant 表。
- 订单与商品：多对多关系，通过 Order_Item（订单明细表）进行解耦。一个订单可以包含多个商品，同时记录购买时的单价和数量。
- 订单与配送员：多对一关系。一个配送员可以承接多个订单，Order 表通过 rider_id 关联 Rider 表。在订单刚创建未分配骑手时，该字段允许为空。

## 五、非功能需求

### 5.1 安全

| 要求       | 实现方式                                                    |
| ---------- | ----------------------------------------------------------- |
| 接口认证   | JWT Bearer Token                                            |
| 权限控制   | RBAC，`IsAuthenticated` / `IsAdmin` 装饰器                  |
| 数据隔离   | ORM 过滤确保商家 / 骑手只能看到自己的数据                   |
| 角色白名单 | 注册接口仅允许 customer / merchant / rider                  |
| 并发安全   | 抢单使用 `select_for_update()` + `transaction.atomic()`     |
| 生产配置   | `settings_prod.py` 强制 DEBUG=False、HTTPS、SECRET_KEY 非空 |

### 5.2 数据完整性

| 要求                | 实现方式                                              |
| ------------------- | ----------------------------------------------------- |
| 下单库存扣减        | 行锁 + 原子操作                                       |
| 取消 / 拒单库存回滚 | `_restore_stock()` 同步回滚 stock 和 sales_count      |
| 起送金额校验        | 下单时服务端检查 `total_amount >= merchant.min_order` |
| 价格校验            | 价格由服务端从商品表读取，不信任前端传入              |
| 默认地址原子性      | `transaction.atomic()` 先清除再设置                   |

### 5.3 性能

| 指标         | 目标                                                        |
| ------------ | ----------------------------------------------------------- |
| API 响应时间 | < 500ms（P95）                                              |
| 并发用户     | 开发阶段不做压测，生产阶段通过 Gunicorn / Daphne 多进程扩展 |
